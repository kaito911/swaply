# Task Template

## 0. 目的

この文書は、SwaplyでAIに実装タスクを渡すときの入力テンプレートを定義する。

目的は以下。

- 人間の指示粒度をそろえる
- AIの初動精度を上げる
- 関連ファイル探索漏れを減らす
- 停止条件を早めに発火させる
- 実装事故を減らす

このテンプレートを使わずに曖昧な依頼をすることは避ける。

---

## 1. 使用ルール

- 毎回このテンプレートをコピーして使う
- 不明な項目は空欄ではなく「不明」と書く
- 禁止範囲は必ず書く
- UI正本参照先は必ず書く
- DBや状態に触れる可能性があるなら明記する
- 危険領域に関係する可能性があるなら先に書く

## 3. 記入例

## タスク名
- propose画面をプロトタイプ準拠に合わせる

## 目的
- 提案画面のUI構成と情報優先順位をプロトタイプ準拠に戻す

## 完了条件
- propose画面の表示順、文言、主要パーツがプロトタイプと一致している
- 型エラーがない
- lintが通る
- 既存の提案フローが壊れていない

## 背景
- 現在の画面はプロトタイプからズレている
- 交換導線の情報優先順位が違う可能性がある

## 変更してよい範囲
- app/(tabs)/propose.tsx
- 関連する表示用コンポーネント
- 型整合に必要な軽微な修正

## 変更してはいけない範囲
- Supabase本番変更
- RLS変更
- offers/tradesの状態遷移変更
- UI/UXの自己判断改善

## 対象画面 / 対象機能
- propose画面
- 提案一覧表示
- 提案状態の表示

## 想定ユーザーフロー
1. 提案タブを開く
2. received / sent / in progress を見る
3. 必要なら承認や確認に進む

## UI正本
- プロトタイプPDF: 該当あり
- 画面仕様書: 該当あり
- プロダクト仕様書: 該当あり
- 該当画面名: propose / 交換提案画面

## 状態正本
- 関係するテーブル: offers, offer_items
- 関係するstatus: pending, accepted, declined など
- 関係する状態遷移: 提案→承認/拒否
- DB確認要否: あり

## 関係しそうな危険領域
- [ ] cards
- [x] offers
- [ ] trades
- [ ] shipments
- [ ] owner_user_id
- [ ] Trust
- [ ] RLS
- [ ] 該当なし

## 既知の関連ファイル
- app/(tabs)/propose.tsx
- lib/supabase.ts
- lib/types.ts

## AIに確認してほしいこと
- 対象画面の関連ファイル
- 型崩れ有無
- UI正本との差分

## AIにやってほしいこと
- 関連ファイル探索
- 影響範囲分析
- 実装
- 型修正
- lint/typecheck前提の整合確認
- PR用記述の整理

## 実装前に止まってほしい条件
- 状態遷移変更が必要な場合
- DB列追加が必要な場合
- RLS変更が必要な場合

## 実機確認で見たいこと
- タブ表示が正しい
- 表示崩れがない
- 提案一覧の導線が壊れていない

## 備考
- 部分コードではなく全文提示
- 推測禁止
- 不明は不明と返す

---

## 4. 最低限必須の入力項目

最低でも以下が埋まっていない場合、AIは実装を開始しない。

- タスク名
- 目的
- 完了条件
- 変更してよい範囲
- 変更してはいけない範囲
- UI正本
- 実装前に止まってほしい条件

---

## 5. AIの返答ルール

このテンプレートを受け取ったAIは、必ず以下の順番で返す。

1. 結論
2. 判断理由
3. 実行手順
4. 必要ファイル / 必要コード（全文）
5. 注意点 / リスク
6. 次にやるべきこと

省略禁止。

---

## 6. 更新ルール

このテンプレートは以下の場合のみ更新する。

- 実運用で入力不足が頻発した
- 危険領域の指定粒度が足りなかった
- UI正本 / 状態正本の参照項目が不足していた
- AIの初動で繰り返しミスが発生した

思いつきで増やしすぎない。
