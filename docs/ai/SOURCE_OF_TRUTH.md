# SOURCE OF TRUTH - Swaply

## 目的

本ファイルは、Swaplyにおける「何を正とするか」を固定するための正式ルールである。

実装、レビュー、仕様判断、修正方針の決定に迷いが生じた場合、
必ず本ファイルの優先順位に従う。

Swaplyでは、
「見た目の正しさ」と「状態の正しさ」は同じではない。

そのため、本ファイルでは
**UI / UX の正本** と
**DB / 状態遷移の正本**
を明確に分けて定義する。

---

## 最上位原則

### 原則1
**UI / UX / 画面構成 / 遷移 / 文言トーンの正本はプロトタイプPDFである。**

### 原則2
**状態 / 所有権 / 成立 / 取引進行 / Trust生成の正本はDBと状態遷移仕様である。**

### 原則3
**現在のコードは正本ではない。**
コードは途中成果物であり、バグや暫定実装を含みうる。

---

## 正本の優先順位

### Layer A: UI / UX / 画面体験の正本

UIや体験に関する判断では、以下の順で優先する。

#### 1. プロトタイプPDF
最優先。
以下を含む唯一の正本とする。

- 画面構成
- 情報配置
- 画面遷移
- UIトーン
- 文言の空気感
- 交換中心の見せ方
- Trustの静かな見せ方

#### 2. 画面単位仕様書
プロトタイプPDFを補足する文書として扱う。

- 各画面の役割
- 画面要素
- 画面間導線
- βで必要な最小画面

#### 3. プロダクト仕様書
画面単位より広い思想を補足する文書として扱う。

- ホームの役割
- 出品・提案・Trustの基本思想
- 情報設計の優先順位
- やってはいけないUI

#### 4. 現在のコード
最後。
コードがプロトタイプとズレている場合、コードを修正対象とする。

---

### Layer B: DB / 状態 / 取引進行の正本

状態や整合性に関する判断では、以下の順で優先する。

#### 1. DBスキーマ / 実DB状態
最優先。
以下を含む。

- status
- owner_user_id
- offer / trade / shipment の実レコード
- adjustment_amount 等の保存値
- 外部キー
- RLS前提の取得可能範囲

#### 2. 状態遷移仕様
DBの意味づけとして優先する。

- offerは提案である
- tradeは成立後取引である
- Trustは完了事実から生成される
- フロントは状態の真実を持たない

#### 3. エンジニア実装仕様書
特に将来拡張や責務分離の基準として扱う。

- venue / supply / hold / trade / trust の責務分離
- DB主導
- atomic処理
- 競合防止
- trade_events 前提

#### 4. 現在のコード
最後。
コードが状態遷移仕様やDB整合とズレている場合、コードを修正対象とする。

---

## 実務上の判断順

### UIの見た目や導線を判断する時
1. プロトタイプPDF
2. 画面単位仕様書
3. プロダクト仕様書
4. 現在コード

### statusや所有権、進行管理を判断する時
1. 実DB状態
2. 状態遷移仕様
3. エンジニア実装仕様
4. 現在コード

### UIとDBが矛盾した時
- 見た目の判断はプロトタイプを優先
- 状態の判断はDBを優先
- つまり「UIを直すのか」「状態ロジックを直すのか」を分けて考える

---

## Swaply固有の絶対原則

### 1. Swaplyは交換中心設計
- 売買ではなく交換がコア
- 双方向性、成立条件、所有権、状態整合を重視する
- 「なんとなく成立」は禁止

### 2. Trustは事実ベース
- 感情レビューを使わない
- 完了事実、発送、返信、トラブルなどの確定事実だけを元にする
- Trustは権利ではなく判断材料

### 3. 判断コスト削減
- UIは情報過多にしない
- 迷いを減らす
- 安心感と軽さを優先する

### 4. 新規救済と流動性
- 一部強者だけが得する構造を避ける
- 差額売買化リスクに常に注意する

---

## 責務分離の正本ルール

以下は混ぜない。

- 商品棚 = 所持 / 潜在供給
- 出品 = 交換対象
- offer = 提案
- trade = 成立後取引
- shipment = 配送進行
- Trust = 完了事実から集計

現在コードがこれを混ぜている場合、コード側が誤り。

---

## 状態の正本ルール

### offer
offerは提案の単位であり、完了の単位ではない。

### trade
tradeは成立後の進行管理の単位であり、完了の単位である。

### shipment
shipmentは配送進行の単位であり、tradeの補助責務である。

### Trust
Trustはtrade完了などの確定事実からのみ生成される。

---

## 禁止判断

以下は禁止。

- UIの表示だけを見て状態を正と判断すること
- ローカルstateを真実とみなすこと
- コードに書いてあるから正しいとみなすこと
- プロトタイプ未照合でUI変更を確定すること
- DB未確認でstatus変更を正当化すること
- offerとtradeを同じ責務として扱うこと

---

## コードの位置づけ

現在のコードは以下の理由で正本ではない。

- 暫定実装を含みうる
- 過去の途中判断が残っている可能性がある
- DB変更や仕様更新に追従していない可能性がある
- UIだけ先に合わせている可能性がある
- 逆に状態だけ先に変えてUIが追いついていない可能性がある

したがって、
**コードは常に監査対象**であり、
正本と一致している時だけ正しいと扱う。

---

## AIの判断義務

AIは修正前に、必ず以下を切り分けて判断する。

### UI判断か？
- 画面構成
- 文言
- 導線
- 情報配置
- 見た目のトーン

→ この場合はプロトタイプを最優先する

### 状態判断か？
- status
- owner_user_id
- offer / trade / shipment の進行
- Trust生成条件
- adjustment_amount の保存

→ この場合はDBと状態遷移仕様を最優先する

---

## 不一致が起きた時の対応

### ケース1: プロトタイプと現在UIがズレる
→ UIコードを修正する

### ケース2: DBと画面表示がズレる
→ まずDBクエリ、型、表示ロジックを監査する

### ケース3: コードと仕様書がズレる
→ コードを修正対象とする

### ケース4: 仕様書同士がズレる
→
1. プロトタイプ
2. 画面単位仕様書
3. プロダクト仕様書
4. 実装仕様書

の順で優先する

### ケース5: DB実態と仕様がズレる
→ まず「DBが間違っているのか」「実装が途中なのか」を保留判定する
→ 推測で確定しない
→ 必要ならSQL確認を優先する

---

## 実装前チェック

重要ファイルを直す前に、以下を確認する。

- これはUIの問題か、状態の問題か
- 正本はどの文書か
- 現DBと一致しているか
- 影響する関連ファイルは何か
- trade / shipment / trust / owner_user_id に波及しないか

---

## 最終ルール

Swaplyでは

- **見た目の正本はプロトタイプ**
- **状態の正本はDB**
- **コードは正本ではない**

この3つを絶対に崩さない。

「動くか」ではなく、
「正本に一致しているか」を最優先とする。
