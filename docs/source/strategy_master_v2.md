この文書は、Swaply に関する現時点の全ての重要事項を統合した**事業母艦**である。単なるプロダクト仕様書ではなく、以下を統合している。

- 事業の本質
- 絶対思想
- プロダクト設計
- Trust 思想
- 成立エンジン
- 差額(調整金)設計
- 配送・住所・安全保障
- 商品棚・推し・文化設計

## 1. Swaply とは何か

### 1-1. 定義

Swaply は、**コレクター向け交換判断インフラ**である。

これは重要で、Swaply は「物を交換するための単機能アプリ」ではない。

Swaply の本質は、**個人間交換における判断コストを構造で削減すること**にある。

より具体的には、**45 秒以内に「この相手と交換してよいか」を判断できる環境を作ること**を目指す。

なお、ここで言う 45 秒は理想価値であり、初回ユーザーの実務上の感覚としてはもう少し長くなる可能性がある。過去の議論で「30 秒」は短すぎるという結論になっており、初回は疑いから入るため 30 秒では現実的でない。したがって、Swaply の理想価値は 45 秒判断だが、β 評価では初回体験について少し緩めに見る余地がある。

### 1-2. 何を解決するのか

現状のオタク交換文化、コレクター交換文化は主に以下のチャネルで成立している。

- X (旧 Twitter)
- LINE
- オープンチャット
- Discord
- リアル交換会
- ライブ会場やイベント現地

そこでは、次の問題が起きている。

**判断コストが高い** — 相手が信用できるかを、DM 履歴や実績ポストやフォロワー数などの曖昧な情報から判断している。

**交渉コストが高い** — 差額、条件、発送方法などを DM で往復するため、交換成立までが遅く面倒。

**トラブルが多い** — 未発送、受取確認遅延、差額トラブル、商品状態相違などが発生する。

**初心者が不利** — 実績がないというだけで不利になり、交換文化に入りづらい。

Swaply はこれを、**Trust 可視化 × 成立エンジン × 安全構造**で解く。

## 2. Swaply の絶対思想

この章は最重要。ここに反する仕様は、たとえ便利でも採用しない。

### 2-1. Swaply は売買アプリではない

Swaply は交換文化のインフラであり、**売買アプリではない**。

したがって、Swaply は常に以下を警戒する。

- レート議論の過剰化
- 差額の高額化
- 売買への流れ
- 価格サービス化
- 転売的利用

Swaply は交換文化を守るため、売買に見える設計を避ける。

### 2-2. 最重要 KPI は成立数

Swaply で最重要なのは MAU ではない。**成立数**である。

理由は明確で、Swaply は密度依存型・流動依存型のサービスだから。

- 登録者が多くても成立しなければ意味がない
- MAU が増えても交換が発生しなければ文化は生まれない
- 最初に必要なのはアクティブの密度と成立体験

したがって、Swaply の議論では、MAU は参考指標に過ぎない。本質は常に**何件の交換が成立したか**である。

### 2-3. 初期流動を作れなければ死亡事業

Swaply は

- 両面市場
- ネットワーク効果型
- 密度依存

の事業であるため、**初期流動が作れない時点で死亡事業**という認識が共有されている。

つまり、どれだけ思想や UI が良くても、**最初の 100 人〜1000 人で密度を作れなければ終わる**。

### 2-4. β の目的は成長ではなく治安証明

β は「ユーザーが増えたか」を見るフェーズではない。β の本質は、

- 治安が保てるか
- Trust 設計が機能するか
- 成立体験を作れるか
- 初心者が成功できるか

の検証にある。

つまり、β の目的は**Swaply が文化を壊さず、交換文化を安全に前進させられるかを証明すること**である。

### 2-5. Trust は権利ではなく判断材料

Trust が高いからといって、

- 後発送できる
- 優先される
- 特別扱いされる

という設計にはしない。

Trust は**相手を判断するための材料**であって、権利ではない。Trust を権利にすると、初心者が詰み、格差が固定し、文化が壊れる。

### 2-6. 交換文化を守りながら、最終的には市場を取りに行く

これはかなり重要な創業者意思である。本音は、思想 100% の小さな世界で終わることではなく、**市場を取りに行くこと**にある。

ただし、市場を取るために最初から思想を曲げるのではなく、**交換文化を守りながら標準を取りにいく**ことが Swaply の基本路線である。

## 3. Swaply の長期ビジョン

### 3-1. Swaply は「世界を変える第一歩」

Swaply はゴールではない。Swaply は、創業者にとって**世界を変える第一歩**である。

勝ちたい気持ちはあるが、それは最終目的ではない。Swaply は「トーナメントの 1 勝」であり、ただしその 1 勝は本気で取りに行く対象である。

### 3-2. 最終形は「個人間信頼取引の基盤」

Swaply の長期構想は、単なるコレクター交換アプリではない。最終的には、**あらゆる個人間信頼取引の基盤**まで視野に入っている。

現時点ではコレクター交換から入るが、長期的には

- 信頼の可視化
- 取引人格の蓄積
- 交換文化の標準化
- 将来的な外部 Trust 連携

まで見ている。

ただし、金融与信やクレカ審査に直結するような話を今の段階で前に出すのは不適切であり、そこはあくまで遠い将来のイメージである。

### 3-3. ムーンショットとしての位置づけ

Swaply は現時点ではまだ「良いニッチ仮説」に近いが、構造としてはムーンショットになり得る。

ムーンショット化する条件は以下。

- Trust 設計がカテゴリ横断で移植可能
- ネットワーク効果が強い
- 成立データが蓄積資産になる
- 文化ごとの違いを吸収しながら共通 OS を作れる

Swaply は現時点では**強い垂直の種**であり、これをどう**水平の OS** に育てるかが長期テーマである。

## 4. 市場と立ち上がり方

### 4-1. Phase 1 ターゲット: 鬼滅 + コナン + サンリオ (v2.2 確定)

Phase 1 β のターゲットは **「鬼滅の刃 + 名探偵コナン + サンリオ」** で確定 (2026-05-09)。

#### 戦略名: 定常型流動性戦略

**理由**:

- **定常型の交換需要** (アニメ・キャラ IP) はスパイク型 (アイドル) と異なり、谷間での離脱リスクが構造的に小さい
- ユーザー実測 X 過去 10 分交換投稿数: アニメ系 12-16 件 vs INI 22 件 (スパイク型)、TREASURE 1 件 — **定常型は底値が高い**
- 規模合算 X 数百万級 (鬼滅 334 万 + コナン 130 万 + サンリオキャラ大賞 6,316 万票)
- 競合空白: トレポータルは 3 候補とも公式リスト未掲載 (Swaply 一番手参入余地あり)
- 創業者現場アクセス可能: 池袋・新宿・渋谷で月複数回 (ufotable Cafe + コナンカフェ + サンリオショップ・ピューロランド)
- 収集型 90% セグメント (Sanrio+ 200 万+、コナン 30 年継続ファン等) を **出品供給源** として活用、交換コア層 1-10% を中核ユーザーに

**v2.1 までの整理 (TREASURE 集中) からの転換**:

旧戦略 (TREASURE/K-POP 集中) は「スパイク型」中心で谷間離脱リスクを抱えていた。Web リサーチ (`memory/project_research_matrix_2026-05-08.md`、38 候補 × 10 軸) と現場ヒアリング (20+ 人、トレポータル認知ゼロ確認) を経て、**定常型 IP 中心のリスク低減戦略** に転換。

**排除候補と理由**:

| 排除候補 | 理由 |
|---|---|
| INI / 国産 K-POP (JO1/ME:I 等) | スパイク型 + ライバル視 (Phase 1 で排除、Phase 2 で再評価) |
| 2.5 次元 (刀ミュ/A3 ステ等) | 規模小 (X 数万-数十万)、3 グループ集中構成での優先度低 (Phase 2 候補) |
| STARTO 主力 (Snow Man/SixTONES 等) | ランダム特典文化なし → Swaply の「ランダム前提交換」設計と相性悪 (Phase 2+) |
| TREASURE | スパイク型 + 規模 (X 580K JP) で定常型 3 候補に劣る |
| SEVENTEEN | トレポータル運営者本人が CARAT で対応確定済 → 一番手不可 |

詳細は `memory/project_research_matrix_2026-05-08.md`、`memory/project_research_sanrio_2026-05-08.md` 参照。

### 4-2. 拡張順 (v2.2 更新)

新ターゲット (アニメ + キャラ IP) を起点とした拡張順:

1. **Phase 1**: 鬼滅 + コナン + サンリオ (定常型 3 候補)
2. **Phase 1.5**: 観察期間 (鬼滅・コナン・サンリオ別の流動性、収集型 → 交換層 onboarding 率を実測)
3. **Phase 2**: 同クラスタ拡張 — 呪術廻戦 / 推しの子 / 薬屋のひとりごと / 葬送のフリーレン (アニメ系)、ハローキティ単独 / クロミ単独 等のキャラ細分化
4. **Phase 2 後半**: 2.5 次元 (刀ミュ/A3 ステ) — ランダム物販密度最強層
5. **Phase 3**: K-POP 国産系 (JO1/INI 等) — スパイク型を定常型と組み合わせて吸収
6. **Phase 3 後半**: STARTO/ジャニーズ系 — ランダム特典なし文化への対応設計を別途検討
7. **Phase 4**: その他グッズ、骨董品 / 美術品など

旧 v2.1 拡張順 (TREASURE → BABYMONSTER → K-POP → ジャニーズ → アニメ) は **定常型優先の構造的判断** に基づき逆順化。アニメ系がリスク低・規模大・競合空白の三拍子のため Phase 1 配置が合理的。

### 4-3. 界隈ごとに文化が違う前提

TREASURE や K-POP のように「推し」が存在する界隈もあれば、

- ポケカ
- 遊戯王
- 骨董
- 美術品

のように、推しではなく

- レアリティ
- 真贋
- 履歴
- 価値

で動く界隈もある。

したがって、Swaply は**アプリを分けるのではなく、モードを切り替える**設計を持つ。

## 5. アプリ構造の大枠

### 5-1. アプリは 1 つ

Swaply はアプリを界隈ごとに分けない。**アプリは 1 つ**。

理由:

- Trust を共通資産にしたい
- 開発コスト・運営コストを抑えたい
- OS 化するには共通基盤が必要

### 5-2. モード切替

ただし文化混在は防ぐ必要がある。そのため、登録断面または利用断面で

- 推し活モード
- トレカモード
- コレクションモード

などの切替を持つ。これは「分断」ではなく**文化レイヤーの切替**である。

### 5-3. Core と Skin の分離

Swaply は以下の二層構造で設計される。

**Core (共通 OS)**

- Trust
- 成立エンジン
- 差額(調整金)設計
- 配送・安全設計
- 提案 → 承認構造

**Skin (界隈ごとに変わる文化)**

- 色味
- 文言
- 推し表示
- 文化トーン
- UI の空気感

これにより、**Trust は共通、文化は界隈ごと**を実現する。

## 6. Trust 設計 (現時点の中核)

### 6-1. Trust の定義

Trust は**取引人格の可視化**である。

Swaply の Trust は単なる「成立しやすさ」「安全度」だけではなく、

- この人は約束を守る人か
- この人は返信が早い人か
- この人は差額に偏りがない人か
- この人はトラブルが少ない人か

といった**取引人格**を表す。

### 6-2. 感情評価は使わない

Swaply では以下を使わない。

- 星レビュー
- コメント欄
- 感情評価
- ランキング
- 上位〇% 表示

理由:

- 人格を壊す
- SNS 化しすぎる
- 序列化する
- 炎上や私怨が入る

Trust は常に**確定事実のみ**で構成する。

### 6-3. 表示する事実

Trust には以下のデータを使う。

- 成立件数 (`trade_count`)
- 発送遵守率 (`ship_rate`)
- 返信中央値 (`reply_median_hours`)
- 受取確認速度
- トラブル件数 (`trouble_count`)
- 差額平均
- 差額偏り
- 最終アクティブ (`last_active_at`)
- 直近アクティブ状況

実装は `lib/types.ts` の `Profile` 型と `computeTrustBadge()` を Source of Truth とする。

### 6-4. Trust は公開型

Trust は**公開型**でいく。

理由:

- 治安が維持されやすい
- 抑止力がある
- DM 流出が減る
- 事前判断が可能になる

### 6-5. 連邦型 Trust

Trust は 2 層構造で考える。

**総合 Trust**

- 本人確認
- 発送遵守
- 返信
- トラブル
- ペナルティ

これは共通基盤。

**界隈 Trust**

- TREASURE 内の成立件数
- K-POP 内の成立件数
- その界隈での差額健全性
- その界隈での返信傾向

つまり**総合 Trust + 界隈 Trust** の連邦型。

ただし、現時点では**界隈跨ぎ (K-POP → ポケカなど) は多くない**という認識も共有されている。Trust 引き継ぎは主戦略ではなく、補助要素レベル。

### 6-6. Trust の外部持ち出し

長期的には、Trust を外部に持ち出したい意図がある。

ただし現実的な第一歩は**外部バッジ (QR やプロフィール表示)** である。クレカ審査などの金融与信は全く別世界であり、今そこを狙う話ではない。

## 7. バッジ設計

### 7-1. バッジの意味

バッジは**信頼の証明**であって、ステータスではない。

つまり、バッジは

- 承認欲求を煽るもの
- 序列を作るもの
- 優遇権を与えるもの

ではない。

### 7-2. 4 段階制 (v2.1 更新)

**現行確定案は 4 段階。** 旧 3 段階 (Bronze / Silver / Gold) は H1 (commit `0ec0f61`) で 4 段階に移行済み。実装は `lib/types.ts` の `computeTrustBadge()` を Source of Truth とする。

| バッジ | ラベル | 条件 |
|---|---|---|
| **Green** | 新規 | 取引 0 件 (デフォルト) |
| **Trial Blue** | お試し | 取引 1 件以上 |
| **Blue** | 安定 | 取引 10 件以上 + 発送遵守率 95% 以上 + トラブル 0 件 |
| **Gold Blue** | 高信頼 | 取引 50 件以上 + 発送遵守率 97% 以上 + 直近 60 日アクティブ |

絵文字や達成度アイコンは使わない。事実の列挙のみ。

### 7-3. 絶対にやらないこと

- 上位表示優遇
- バッジランキング
- 上位〇% 表示
- バッジ限定機能
- バッジの有料化 (課金可否は **章 13-1 の 4 分類ルール**を参照)

### 7-4. 外部展開

最初の外部展開はバッジの外部持ち出し。

例:

- X プロフィールに貼れる
- QR 表示できる
- 交換会で見せられる

ここが現実的な第一歩。

## 8. 成立方式

### 8-1. Swaply は提案 → 承認型

成立方式は **提案 → 承認**で確定。マッチングアプリ型の「双方いいね → 自動成立」は採用しない。

### 8-2. 理由

- マチアプ感が強く、文化デメリットになり得る
- 交換は条件主導であり、感情消費ではない
- 成約率が下がる
- カード、差額、Trust など条件提示が必要

したがって、Swaply にとって自然なのは**A が提案し、B が承認する**という構造。

### 8-3. 相手側の選択肢

提案を受けた側は

1. 承認
2. 差額変更提案 (カウンター提案として扱う)
3. 辞退

差額変更提案は「編集」ではなく**カウンター提案**として扱う。

### 8-4. 提案数

提案数は**無制限**。

理由:

- 制限する意図がない
- 初期流動を殺したくない
- Swaply は制限で秩序を作るのではなく、構造で秩序を作るサービス

ただし、内部では

- 承認率
- 条件不一致率
- 短時間大量送信

などを見て、Trust や露出に反映する余地がある。

## 9. 成立エンジン (Success Engine)

### 9-1. 成功体験が生死を決める

Swaply で最も重要なのは、**初回成功体験をどれだけ早く作れるか**である。

登録しても成立しなければ、

- 結局 DM の方が早い
- 過疎だ
- 使う意味がない

で終わる。

したがって、Swaply の真のコアは**成功体験エンジン**にある。

### 9-2. レーン構造

ホームは 3 レーン。

- **レーン 1**: おすすめ
- **レーン 2**: 成立しやすい交換 → これが最重要
- **レーン 3**: 新着

H4 (commit `298fa54`) で Smart レーン (レーン 2 推薦アルゴ) は β 開始時の暫定状態として placeholder 化済み。本実装は Phase 2 backlog。

### 9-3. レーン 2 の役割

レーン 2 は**初心者の成功体験を作る装置**である。

ここで

- 初心者でも成立する
- Trust がたまる
- 継続する

という循環を作る。

### 9-4. β では半人力運用

成立エンジンは **半人力**が最適と判断されている。

理由:

- 初期はアルゴが弱い
- データ不足
- 立ち上がりで密度が重要

したがって、β では

- アルゴ
- 人力キュレーション

を併用する。

### 9-5. 優先ピックの考え方

レーン 2 では以下が優先される。

- 初心者
- 人気メンバー
- 差額なし
- Trust 差が小さい
- ログイン中
- 成立しやすい組み合わせ

## 10. 商品棚・推し・おすすめエンジン

### 10-1. 目的

商品棚と推しは、Swaply に

- 登録理由
- 継続理由
- 所属感
- 文化性

を与える。

### 10-2. 商品棚

商品棚は**完全任意**。

商品棚は作ってもよいし、作らなくてもよい。作らないことによるデメリットは作らない。

つまり

- UX を壊さない
- ヘビーユーザーには強い武器になる

という位置づけ。商品棚を作ると、**成立可能性の高い交換提案が出やすくなる**というメリットがある。

### 10-3. 推し

推しは登録できる。ただし、**推しを数値化しすぎない**。

理由は、オタク文化では「推しは神聖」「推しは尊い」という感情があり、それをデータ化・ランキング化しすぎると嫌悪感が出るため。

したがって、推しは

- フィルター
- 文化レイヤー
- 世界観スイッチ

として使う。

### 10-4. 推しの役割

推しは

- テーマカラー
- 優先表示
- 推しに合う交換のおすすめ
- マイページの所属感

に使う。Trust や市場価値とは切り分ける。

### 10-5. おすすめエンジン

おすすめは、

- 商品棚
- 出品
- 推し
- Trust
- 一致度

から生成される。

特に強い仮説は、**自分の商品棚の所持と相手の求が一致 + 相手の出品商品が自分の推しに一致**の時に、おすすめとして強く出す。

この設計により、

- 登録率
- 成立頻度

の両方が上がる可能性が高い。

## 11. Swaply は所属型サービスである

### 11-1. 結論

Swaply は**所属するサービス**を目指す。

便利な交換ツールではなく、**界隈に所属する感覚を持てるサービス**を狙う。

### 11-2. 便利型ではなく所属型である意味

所属型になると、

- 用がなくても開く
- 離脱しづらい
- LTV が伸びる
- アクティブ率が上がる

Swaply では

- 推し
- 商品棚
- 文化モード
- 交換会との接続

がこの所属感を生む。

### 11-3. 最終形の構造

最終形は**文化ごとに強い小さな国の集合体**である。

つまり、Swaply は

- TREASURE 文化
- K-POP 文化
- ジャニーズ文化
- 骨董文化

をそれぞれ持ちながら、**Trust という共通基盤を持つ文化連邦型サービス**を目指す。

## 12. 差額(調整金)設計

### 12-1. 差額の定義

差額は**売買ではなく調整金**である。

使う言葉も固定。

- 支払い → 使わない
- 決済 → 使わない
- 購入 → 使わない
- **調整金** → 使う
- **仮確保** → 使う

### 12-2. 差額は任意

差額文化がある界隈もあるので、差額は**任意**とする。

ただし、心理的に使いにくい設計にはする。

- 任意だが推奨しすぎない
- 文化としては差額なしが自然に見える

### 12-3. 差額はデフォルト非表示

交換提案画面で差額欄は最初から出さない。必要な時だけ開く。これにより、差額文化を前面に出さない。

### 12-4. 差額上限

差額上限は **¥1,000**。料金プラン全 6 種で共通 (章 13-2)。

理由:

- 売買化防止
- トラブル防止
- 交換文化の維持
- UI の説明容易性

### 12-5. 差額入力者

差額は**提案者のみ入力可能**。

理由:

- 双方が編集可能だと収拾がつかない
- UX が複雑になる
- 交渉が長引く
- 提案 → 承認構造を守りたい

実装は E2 (commit `0b32a50`) で `adjustment_amount` を end-to-end 永続化済み。符号エンコードで方向 (proposer 払い / receiver 払い) を表現。

### 12-6. 相手側の対応

受け手は

- 承認
- 差額変更提案
- 辞退

ができる。差額変更提案は新しいカウンター提案として扱う。

### 12-7. 差額エスクロー

差額がある場合の流れ:

1. 交換成立
2. 調整金を仮確保
3. 双方発送
4. 受取確認
5. 解放

エスクロー実装本体は Phase 2 backlog。詳細は `pricing_v1.md` を参照。

### 12-8. 売買化防止ルール

差額だけの取引は不可。必ず交換対象のカードやグッズが存在することが前提。

### 12-9. Trust との関係

差額は人格指標でもある。

- 平均差額
- 差額偏り
- 常に相手に出させる傾向

などは Trust に反映しうる。

## 13. 料金プラン (確定版 v1)

### 13-1. 課金可否の 4 分類ルール (v2.1 追加)

Swaply の課金体系は、機能を以下の 4 分類に整理し、Trust 系・Match 系は**課金不可**、Information 系・Convenience 系は**課金可**とする原則を持つ。

| 分類 | 内容 | 課金 | 理由 |
|---|---|---|---|
| **Trust 系** | バッジ、信頼スコア、相手評価 | **不可** | 信頼の貨幣化を防ぐ |
| **Match 系** | おすすめ、出会い、出品露出 | **不可** | 出会いの貨幣化を防ぐ |
| **Information 系** | 密度可視化、市場情報、Trust 詳細 | 可 | 判断材料の付加価値 |
| **Convenience 系** | 通知優先度、UI 改善、利便性 | 可 | 体験の付加価値 |

この原則により、Swaply は「**金で信頼を買えない**」「**金で良い相手を引けない**」構造を維持する。Premium との差は安全ではなく、利便性と情報粒度。

### 13-2. プラン構成 (v2.1 更新)

旧版の 2 プラン構成 (Standard ¥300 / Premium ¥600) を、以下の 6 プラン構成に拡張。

| プラン | 価格 | 差額手数料 |
|---|---|---|
| **Free** | ¥0 | 5% |
| **Standard 単月** | ¥300 / 月 | 5% |
| **Standard 3 ヶ月パス** | ¥800 / 3 ヶ月 | 5% |
| **Standard 通年** | ¥3,000 / 年 | 5% |
| **Premium 3 ヶ月パス** | ¥1,500 / 3 ヶ月 | 3% |
| **Premium 通年** | ¥6,000 / 年 | 3% |

全プラン共通: **調整金上限 ¥1,000** (章 12-4)。

詳細な課金フロー、エスクロー処理、税抜表示、解約ポリシーは [`pricing_v1.md`](./pricing_v1.md) を Source of Truth とする。

### 13-3. 基本思想

Swaply は基本利用を Free で許容しつつ、定常利用ユーザーには **Standard 以上を推奨**する構造を持つ。

- Trust / 治安維持 / エスクロー / 不正対策 / 文化維持にコストがかかる
- ライト層は Free で初回成功体験を経て、定着後に Standard へ
- 中長期パス (3 ヶ月・通年) は早期コミットを促す

旧版の「**インフラ利用料を全員が払う**」原則は、長期目標として保持。当面は Free でライト層を取り込み、定着後に課金導線を引く 2 段階アプローチ。

### 13-4. Standard と Premium の差

| 軸 | Standard | Premium |
|---|---|---|
| 安全性 | 同等 | 同等 |
| 差額手数料 | 5% | 3% |
| 成立可能性表示 | 標準 | 高度 |
| Trust 詳細 | 標準 | 高度 |
| おすすめ | 標準 | 高度 |
| 利便性 | 標準 | 強化 |

Premium の価値は「安全」ではなく「利便性・情報粒度」。これは 13-1 の 4 分類ルールに従い、Trust / Match 系は無償平等、Information / Convenience 系のみ課金対象という原則の現れ。

### 13-5. 危険ポイントとフェーズ設計

最初から全員が課金前提だと、初回成功体験前に課金されるリスクがある。

- **フェーズ 1: β** — 完全無料 (`FEATURE_FLAGS.BILLING = false` で構造的保護、H5 commit `c145018`)
- **フェーズ 2: β 後** — Free プラン解放、Standard / Premium は任意
- **フェーズ 3: 密度が十分できた後** — Standard 推奨を強化、Free はライト層のみ

`FEATURE_FLAGS` は β で誤って課金フローが走らないようにするための構造的安全装置。

## 14. 配送設計

### 14-1. 発送順は完全平等

Trust で発送順を変えない。

理由:

- UX を壊す
- 初心者を不利にする
- Trust 格差固定
- 文化破壊

### 14-2. 発送期限

交換成立後、**72 時間以内に発送**。

### 14-3. 未発送時

72 時間以内に発送されない場合

- 自動キャンセル
- 差額(調整金)は返金
- Trust 減点

72h 自動キャンセル実装本体は Phase 2 backlog。

### 14-4. 片側だけ発送した場合

理想は**配送を途中で止めて商品を戻せる構造**。

ただし、配送会社仕様や β 運営コストを踏まえ、初期は

- 配送停止方法の案内
- Trust ペナルティ
- 72h ルール

で対応する。

### 14-5. 追跡番号

発送時には**追跡番号必須**。

理由:

- 発送証明
- 詐欺抑止
- Trust 蓄積

追跡番号必須化と Trust 反映は Phase 2 backlog。

## 15. 住所・個人情報設計

### 15-1. β では匿名配送なし

匿名配送は将来的にはやりたいが、β では採用しない。コストと提携が重いから。

### 15-2. 交換成立前は完全非公開

住所、名前、電話番号は交換成立前には非公開。

### 15-3. 交換成立後のみ配送情報表示

交換成立したら、配送に必要な情報のみ表示。

- 配送用名前
- 郵便番号
- 住所

### 15-4. 電話番号は相手に公開しない

## 18. 18 ヶ月構想と勝負トリガー

### 18-1. 全振りトリガー

全振りトリガーの最重要条件は**月間成立 1.5 万件**。

理由:

- 1 万件で界隈制覇に近い
- 1.5 万件で横展開の見通しが立つ
- 文化浸透の兆候が出る

### 18-2. 期間目標

月 1.5 万件到達目標は **18 ヶ月**。

1 年では界隈制覇は難しい。2 年目のツアーで、交換会で「Swaply」という言葉が自然に飛び交う状況を理想とする。

### 18-3. 構造分解

月 1.5 万件は

- アクティブ 6,000 人
- 平均 2.5 回 / 人 / 月

のモデルで見る。登録は 15,000 人規模が必要という認識。

### 18-4. 賭ける条件は感情ではなく構造

全振りは感情でやらない。条件でやる。現在の重要条件は少なくとも以下。

- 月間成立 15,000 件以上
- LTV/CAC > 3
- 横展開の再現性
- DM 流出率が低い
- 文化浸透

### 18-5. Phase 構造 (v2.2 更新)

v2.2 で Phase 1 ターゲットを TREASURE/K-POP から **アニメ + キャラ IP (定常型流動性戦略)** に転換 (章 4-1 参照)。

- **Phase 1: 定常型流動性確立期** (β 開始 2026-06 想定) — **鬼滅の刃 + 名探偵コナン + サンリオ** 集中、収集型 90% セグメントを出品供給源、交換コア層を中核ユーザーに onboarding。創業者前面戦略 (池袋・新宿・渋谷の現場アクセス) を最大化
- **Phase 1.5: 観察期間** — 鬼滅・コナン・サンリオ別の月成立件数、収集型 → 交換層転換率、コラボ需要 (鬼滅 × サンリオ等の IP × IP) を実測。次手 (Phase 2 拡張作品の選定) を決定
- **Phase 2: 同クラスタ横展開期** — 呪術廻戦 / 推しの子 / 薬屋のひとりごと / フリーレン (アニメ系) → 2.5 次元 (刀ミュ/A3 ステ) → K-POP 国産系の順
- **Phase 3+**: ジャニーズ系 (STARTO ランダム特典なし対応設計) → その他グッズ → 骨董/美術品

旧 v2.1 構造 (Phase 1 = TREASURE 集中、Phase 1.5 = INI 再検討、Phase 2 = BABYMONSTER → 他 K-POP → アニメ) は **「規模単一軸での議論」に偏り定常型 vs スパイク型の構造的視点を欠いていた**。v2.2 で逆順化。

戦略 v3 (`strategy_v3.md`) は v2.2 ターゲット転換に伴い更新が必要 (TREASURE 集中・INI 再検討・観察者ポジションの記述箇所が新ターゲットと不整合)。次回戦略 v3 改訂時に整合化予定。

## 20. 現時点での成功率評価 (v2.1 更新)

直近の主観評価は、**4 シナリオ確率分布**として整理した。旧版の 3 シナリオ評価 (成功 35% / 並成功 45% / 失敗 20%) から、4 シナリオ + 数値見積もりに精緻化。

### 20-1. シナリオ確率分布

| シナリオ | 確率 | 内容 |
|---|---|---|
| **失敗** | 5–8% | 初期流動が立ち上がらず、成立密度が形成されない |
| **並成功** | 30–35% | β 達成、限定的な界隈定着 |
| **成功** | 40–42% | 月間成立 1.5 万件到達、横展開の見通し成立 |
| **大成功** | 17–23% | 文化浸透、複数界隈で標準化 |

### 20-2. 期待値

- **Year 3 期待純利益**: 約 ¥8,500 万 〜 ¥1.05 億
- **創業者リターン期待値** (税引後・Bootstrap + Series A 想定): 約 ¥1.7 〜 2.1 億

### 20-3. 精緻化の理由

- 成立数重視に一本化された
- 賭けの条件が数値化された (章 18-4)
- Trust = 人格まで昇華した
- 商品棚 + 推し + おすすめで登録理由と成立頻度の仮説が強くなった
- 初期流動の重要性を直視している
- **料金体系 (章 13)、Phase 構造 (章 18-5)、4 分類ルール (章 13-1) が実装可能な解像度まで詰まった**
- M5 (CI 強制) で型レベルの再発防止が構造化された

ただし、これはあくまで**設計フェーズにおける確率改善**であり、現実世界ではまだユーザー 0、成立 0 なので、**実行力が最大変数**。
