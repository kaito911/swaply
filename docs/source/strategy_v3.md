## 1. 前提

### 1.1 創業者ポジション: 観察者

Swaply の創業者は TREASURE のファンではない。BABYMONSTER も同様。INI には人脈接点があるが戦略上は使わない(後述 2.2)。

これは弱みではなく**設計条件**として扱う。観察者前提の戦略は以下を含意する:

- ファン心理の一次的解像度はコミュニティから取る(β招待者、主催者ヒアリング、X投稿の継続観察)
- 推しの気持ち代弁による意思決定をしない。ファン代表ではなく構造設計者として振る舞う
- 「自分が欲しいから作る」ロジックに依存しない。数字とユースケースで判断する

逆に、観察者だからこそ取れる態度:

- どの界隈にも肩入れせず、Trust 思想・調整金ルール・4分類ルールを界隈横断で一貫適用できる
- ファン感情のノイズを拾わず、構造の正しさを優先できる

**観察者の限界対処**:

- β招待者の選定で「界隈解像度の高い主催者」を優先する
- 主催者からのフィードバックを構造的に収集(月次のヒアリング等)
- 「自分の判断で迷ったら主催者に聞く」を意思決定ルールにする

### 1.2 対象選定はデータドリブン

K-POP 男性アイドルグループから TREASURE を選定した根拠:

- X/Twitter 上のトレカ交換投稿の密度(観測時点で他候補を上回る)
- 2026年7月のライブが β 開始(6月)直後にある = 主催者ハイジャックの好機が制度的に存在
- 4世代(ENHYPEN, TXT, Stray Kids 等)と比較して、ファンダム規模に対するアプリ未成熟領域が大きい

注: これらは観測時点の仮説であり、β 期間中に検証する。

### 1.3 ハードデッドライン

- β 開始: 2026年6月
- TREASURE ライブ: 2026年7月

ライブ会場は最大の流動性発生源。Phase 1 完了は**この1回のライブ周期**に賭けて評価する。

## 2. 対象選定

### 2.1 中核: TREASURE

- Phase 1 / Phase 1.5 / Phase 2 の主戦場
- 全機能仕様(Trust、調整金、4分類)は TREASURE 文脈で第一実装
- 主催者ハイジャック戦略(章3)の対象界隈

### 2.2 戦略除外: INI

INI は K-POP 隣接の有力候補だが、Phase 1 戦略から除外する。

**除外理由**:

- **(a) 創業者と INI メンバーとの個人的接点**: 連絡可能だが多忙で非継続的。戦略要素として扱うと、連絡が取れない期間に意思決定が滞る
- **(b) 創業者の個人的接点リスク**: INI 界隈には創業者の個人的関係に関わる要素があり、能動的活動が判断の客観性を損なうリスクがある
- **(c) 観察者の清浄性**: 観察者として K-POP 全体を扱うなら、(a)(b) のような個人的紐づけがない界隈の方が判断が清浄

> **L(Devil)**: 個人事情で戦略を狭めるのは合理的か。  
> **A(Strategy)**: 機会費用は発生する。ただし (a) の連絡非継続性は構造的問題で、運頼みの戦略は再現性がない。除外は合理。

### 2.3 観察対象: BABYMONSTER

- TREASURE と同じ YG 所属、女性グループ
- Phase 2 以降の横展開先候補として、β 期間中も観測を継続
- 主催者ネットワークの一部が TREASURE と重複する可能性。Phase 1 で得た主催者関係を Phase 2 に橋渡しできる余地あり

### 2.4 戦略外: 他の K-POP / J-POP

ENHYPEN、TXT、Stray Kids 等は密度・接続性で TREASURE に劣るか、創業者の解像度が薄い。Phase 2 までは扱わない。

## 3. Phase 1: 主催者ハイジャック

### 3.1 戦略の核

ライブ前後のトレカ交換は、ファンダムごとに**特定の主催者**(X 上のアカウント)が交換会・譲渡まとめを取り仕切っている。彼らに Swaply を採用させることが Phase 1 の唯一の目標。

理由:

- ファン個別獲得は CAC が高い
- 主催者経由なら、彼らがすでに持つフォロワー(数千〜数万)に1回のリーチで届く
- 主催者の「お墨付き」は、Trust 概念そのものの社会的実装になる(Trust 思想と整合)

### 3.2 主催者の定義

以下のいずれかを満たす X アカウント:

- TREASURE 公演ごとに「交換会」「譲渡まとめ」を主催している実績がある
- フォロワー 1,000 以上 かつ トレカ関連投稿が継続的
- 過去のライブ周期で主催活動が確認できる

### 3.3 Phase 1 の動き

1. **主催者リストアップ**(β開始前、5月中)
   - X 検索で TREASURE 公演会場名 + 「交換」/「譲渡」で抽出
   - 25〜50名を初期ターゲットに
2. **β招待**(6月初旬〜)
   - 5〜10名を先行招待、無料で全機能利用可
   - フィードバック収集と機能修正の主戦場
3. **主催者主導の集客**(6月後半〜7月)
   - 招待した主催者が彼らのフォロワーを誘導
   - 創業者は黒子、Swaply の存在は主催者経由で広まる
4. **ライブ会場での実運用**(7月)
   - 会場周辺の交換需要を Swaply で受ける
   - Trust バッジが「会場で会っても安心」の根拠になるかを検証

### 3.4 創業者の役割

- 主催者との関係構築(DM、必要なら通話)
- フィードバックの即時実装(Claude Code との作業を継続)
- データ観測(DAU、交換成立率、調整金利用率)

ファン向けマーケティングは行わない。主催者だけを見る。

## 4. Phase 1 完了基準

Phase 1.5 への移行は以下のすべてを満たした時点で判定する。機械的に判定可能であることを優先する。

### 4.1 定量基準

- **主催者採用数**: 5名以上が継続使用(ライブ後2週間にわたり最低1回以上のアクティブ)
- **成立交換数**: TREASURE ライブ周期(2026年7月〜8月)で合計100件以上の交換成立
- **Trust バッジ取得者**: Blue 以上が 5名以上(Blue = 取引10件 + ship_rate 95% + trouble 0、主催者の運用品質の証拠)

### 4.2 定性基準

- 主催者から「次のライブでも使う」の言質が3名以上
- 重大バグ・信頼毀損事象が発生していない
- 4分類ルール違反(課金不可機能で課金が発生する等)が0件

### 4.3 未達時の判断

定量1つ以上未達 → Phase 1 延長(次のライブ周期まで継続観測)。Phase 1.5 には進まない。

> **B(Economist)**: 100件は控えめ。主催者5名 × 各20件で達成可能。これより低いと「動いた」と言うには弱い、これより高いとライブ1回では未達リスクがある。  
> **L(Devil)**: 数字だけで判断すると「動いてるけど質が悪い」を見逃す。定性基準(特に重大バグ0)を必ずチェックすること。

## 5. Phase 1.5: INI 再検討

Phase 1 完了基準を満たした時点で、INI 戦略への展開を再検討する。

### 5.1 再検討する理由

- INI は K-POP 隣接の有力候補で、Phase 1 で除外したのは個人事情ベース
- TREASURE で構造が動くことが証明できれば、個人事情を回避する設計(後述)が成立する余地がある

### 5.2 再検討時に決める論点

- **INI メンバーとの個人的接点は使うか**: 原則として使わない(連絡の非継続性は構造的)。ただし当該メンバー側から能動的接触があれば検討
- **個人的関係に起因する接触リスクをどう扱うか**: 創業者が直接動かず、招待した主催者経由のみで広める運用にする
- **既存 TREASURE インフラとの統合度**: 別界隈として完全分離するか、Trust バッジを共通化するか

### 5.3 Phase 1.5 で先送りすること

- INI 専用の機能差分は作らない(Phase 1 で確立した仕様を流用)
- 新規市場の検証として動かす、構造変更は含めない

## 6. Phase 2 以降: 横展開

### 6.1 横展開の順序(仮説)

> 注: INI は Phase 1.5 で別フローとして再検討する(章5)。本章の通常順序には含めない。

1. **BABYMONSTER**(YG 同社、主催者重複の可能性)
2. **他 K-POP**(ENHYPEN, TXT 等、密度の高い順)
3. **アニメ・ソシャゲグッズ**(トレカ交換の隣接市場、ユースケース転用可能性)

### 6.2 横展開の判断基準

- 既存界隈の密度が「成立交換数の伸び率」で頭打ちになった時に次界隈を開く
- 並行運営できる主催者リソース(招待・サポート)が確保できているか
- 4分類ルール・Trust 思想が界隈差なく適用できるか

### 6.3 アニメ・グッズ展開の前提

K-POP と異なり、ライブ会場のような時空集中型イベントが少ない。代わりに:

- イベント(コミケ、オンリー)期間に合わせた周期運営
- グッズ発売周期(新弾発売)に合わせた在庫流動の山

を見立てる必要がある。Phase 3 で詳細設計する。

## 7. 戦略要素の位置づけ

機能仕様の根拠を戦略レベルで再確認する。実装詳細は別文書(事業母艦v2、料金確定版v1、改修指示書v1.1)。

### 7.1 Trust 思想

**バッジは信頼証明であって序列ではない。**

- 4段階(Green / Trial Blue / Blue / Gold Blue)は「達成度」ではなく「観測された信頼の累積」を示す
- ユーザーが「上を目指す」よう設計しない。「自分の活動が事実として記録される」状態を作る
- mypage 表現は事実列挙のみ(H1 で実装済み)。励まし・煽り・進捗バーは入れない

戦略的含意:

- Trust は主催者ハイジャック戦略と表裏。主催者が高 Trust 状態であることが、彼らのフォロワーへの推薦の根拠になる
- 序列化すると、主催者 vs 一般ユーザーの非対称が「ランク差」と誤読される。それは Trust の意味を壊す

### 7.2 調整金 ¥1,000 上限

**「差額」ではなく「調整金」と呼称し、上限¥1,000を設ける。**

- 「差額」は売買のニュアンスを帯びる、調整金は交換のニュアンスを保つ
- ¥1,000 上限は、交換が売買に滑り落ちるのを防ぐ構造的ガード
- 上限超過の取引は4分類上「課金不可」になる設計(料金確定版v1 参照)

戦略的含意:

- 単価を上げて手数料収益を最大化する誘惑を、上限で構造的に遮断している
- これは短期収益と引き換えに「Swaply は交換のためのもの」という界隈認識を買っている

### 7.3 4分類ルール

機能を Trust / Match / Information / Convenience の4分類に分け、課金可能性をガードする。

- **Trust 系**: 課金不可(信頼の貨幣化を防ぐ)
- **Match 系**: 課金不可(出会いの貨幣化を防ぐ)
- **Information 系**: 課金可(密度可視化、市場情報)
- **Convenience 系**: 課金可(通知優先度、UI 改善)

戦略的含意:

- 課金可能性を構造で縛ることで、収益化フェーズで思想がブレない
- FEATURE_FLAGS(H5 で実装)が運用上のガード、4分類ルールが思想上のガード
