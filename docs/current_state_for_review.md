## 1. アプリ概要

**Swaply** (スワプリー) は、推し活グッズ (アイドルのトレーディングカード、アクリルスタンド、アニメキャラのフォトカード等) を **ユーザー同士で交換** する日本市場向けスマホアプリ。販売ではなく「交換」に特化している点が独自。

対象ユーザー: 鬼滅の刃・名探偵コナン・サンリオなどのアニメ／キャラ IP ファン、および K-POP (TREASURE, SEVENTEEN 等) のファン。年齢層は 10 代後半〜30 代の女性中心。

解決する課題: 現状の主な交換手段は X (旧 Twitter) の DM で「譲ります／求めます」ポストをやり取りすることだが、検索性が低い・相手の信頼度が不可視・トラブル時の救済がない、という摩擦が大きい。Swaply はこれを構造化された場で解決する。

## 2. 差別化の 3 本柱

### 2.1 求⇄譲の並行検索 (「マッチ」機能)
検索画面に 3 つのモードがある:「**譲を探す**」(欲しい商品で出品を検索)、「**求を探す**」(自分が手放したい商品を欲しがっている人を検索)、「**マッチ**」(自分が出す商品 + 自分が欲しい商品の両方を入力して、お互いの希望が一致する相手を一発で表示)。最後のマッチは X の DM では実現できない、Swaply の核機能。

### 2.2 Trust (トラスト) 表示
ユーザー間の信頼度を**事実データのみ**で 4 階層に分類して表示する仕組み: **新規 / お試し / 安定 / 高信頼** (アイコン色は順に Green / Teal / Sky / Amber)。指標は取引成立件数・発送遵守率・返信中央値・トラブル件数等、すべて事実ベース。星評価や感情レビューは採用しない (序列化と人気投票化を構造的に防ぐため)。

### 2.3 会場モード
ライブ会場・コラボカフェ等の同一会場にチェックインしたユーザー同士で**現地手渡し交換**を促進する機能。Swaply は現地での手渡し交換を一級機能として正式サポートしている (会場一覧 / チェックイン / 同一会場ユーザー表示 / 当日交換予約 = Venue Hold)。会場での即時マッチングは推し活ユーザーの実利用シーン (ライブ前後・コラボカフェ等) と整合。

## 3. 確定済みの設計判断 (外部レビューで蒸し返し不要)

以下は意図的に決めており、議論済みの設計原則:

- **「交換」であり「売買」ではない**: 取引時の差額は「差額」ではなく「**調整金**」と呼称、上限 ¥1,000 で DB レベル制約。単価上昇による手数料収益最大化の誘惑を構造的に遮断する。
- **Trust の貨幣化禁止**: Trust 表示・バッジは課金対象にしない。信頼を売り買いさせると Trust 思想が崩壊するため。
- **機能の 4 分類ルール**: Trust 系・Match (マッチング) 系は課金不可。Information (情報可視化) 系と Convenience (利便性) 系のみ将来課金候補。
- **完全無料コア + 段階的収益化**: ベータ期は完全無料。Phase 2 以降に取引時の差額決済手数料 10%、Phase 2.5 で控えめ広告、Phase 3 で軽い任意プラン (Plus、月 ¥300-400)、Phase 4 (法人化後) で B2B データ提供。コア機能 (出品・検索・チャット・Trust・会場モード) は永続無料。
- **著作権配慮**: 実在 IP のロゴ・キャラ画像はアプリ内マスタに保持しない。ユーザー出品画像は自撮り強制、自動リサイズで著作権法 47 条の 2 政令準拠 (32,400 px 以下)。「○○風」「○○系」記載は規約禁止 (メルカリ業界標準準拠)。
- **顔出し・自分の声 NG** (運営側): 運営は匿名方針 (顔・声を出さない)。SNS は中の人スタイル + AI ナレーション。
- **Pioneer 制度**: ベータ期初期協力者 50 名限定の特典制度 (バッジ永久 / 広告非表示永久 / 差額決済手数料 3 年無料 + 段階逓減)。希少性最大化・後発ユーザーへの心理影響最小化。

## 6. 技術構成

React Native + Expo (ファイルベースルーティングの Expo Router) で iOS/Android のクロスプラットフォーム実装。TypeScript で型安全に。バックエンドは Supabase (PostgreSQL + Auth + Storage + Row Level Security) を採用、サーバーレスで個人開発に整合。検索パフォーマンスのため pg_trgm (PostgreSQL 拡張) を利用したファジーマッチを準備中。画像配信は Supabase Storage、画像表示は expo-image (キャッシング + フェードイン)。決済は Phase 2 で Stripe Connect (収納代行スキームで資金移動業登録を回避) を予定。

## 7. 既知の課題・未解決の論点

- **マッチング検索のスケーラビリティ**: 「マッチ」モードは現状クライアント側で 2 段階 JOIN している (cards テーブルと wanted_cards テーブルを別々に取得して JS で突合)。DAU 数千規模までは問題ないが、Phase 2 以降は Postgres function (RPC) 化が必要。
- **wanted_cards (需要 DB) のテキスト紐付け脆性**: 現状 wanted_cards は card_id 列を持たず、商品名のテキスト fuzzy match で出品とのマッチングを判定している。構造化カードが導入された後もテキストで紐付けるため、表記揺れ・改名で取りこぼしが出る。card_id 列追加と matcher v3 化を予定。
- **広告とブランドポジショニングの緊張**: ブランド方針は Aēsop / Linear / Notion 系の「信頼 × おしゃれ」象限。AdMob バナーは構造的に相性が悪い。ネイティブ広告 (公演グッズ告知のみ) に絞る方針だが、Phase 2.5 着手時に再検証が必要。
- **K-POP master 化の遅延**: アニメ系 (鬼滅・コナン・サンリオ) は master_characters を整備済だが、K-POP は legacy のテキスト 3 段検索が残っている。Phase 1.5 (2026/9 INI ドームツアー期) までに master 化を完了させる計画だが、データ準備工数の見積もりが緩い。
- **IP 弁護士相談がベータ前未完了**: 創業者のスケジュール都合で対面相談の予約が取れず、メール相談 (弁護士ドットコム) で代替予定。サンリオは特に厳格 (商用は個別ライセンス審査) のため、ベータ前に法的判断が出ない可能性。
- **Pioneer 制度の「越えられない壁」リスク**: 50 名限定 = 残り 1 億 2 千万人は永久に Pioneer になれない。後発ユーザーへの心理影響を最小化するため「ベータ期間限定の創業期協力者」と明示しているが、コミュニティ拡大時に再評価が必要。
- **マッチ (3 モード目) の検索 UX**: ユーザーが「自分の出す商品名」を正確に入力する負担が大きい可能性。自分の出品から選択する UI を後続フェーズで検討。
- **会場モード初期流動性**: 会場機能は機能的に強みだが、ベータ初期 (DAU 100 規模) では同一会場に複数ユーザーが集まる確率が低い。初期は「会場での即マッチング」訴求が空回りする可能性。
