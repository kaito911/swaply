# PR REVIEW CHECKLIST

## 0. 目的

このチェックリストは、SwaplyにおけるPRレビュー時の判断基準を統一するためのもの。

人間はこのチェックをすべて通過した場合のみPRを承認する。

---

## 1. 前提

* このPRはAIによって作成されている
* AIはミスる前提でレビューする
* 「動く」は合格条件ではない

---

## 2. 変更概要チェック

* [ ] タスク名が明確
* [ ] 目的が明確
* [ ] 完了条件が具体的

---

## 3. Source of Truthチェック

### UI

* [ ] プロトタイプと一致している
* [ ] 勝手なUI変更がない

### 状態 / データ

* [ ] DB構造と矛盾していない
* [ ] 状態遷移を壊していない

---

## 4. 対象 / 関連ファイルチェック

* [ ] 対象ファイルが正しい
* [ ] 関連ファイルの読み漏れがない
* [ ] 不要なファイル変更が含まれていない

---

## 5. 影響範囲チェック

### UI

* [ ] 他画面に影響していない

### DB

* [ ] DB変更が含まれていない（含まれる場合は別PR）

### 状態遷移

* [ ] 状態ロジックが壊れていない

### RLS

* [ ] RLSに影響していない

### ルーティング

* [ ] 画面遷移が壊れていない

### 型定義

* [ ] 型崩れがない

### 既存機能

* [ ] 既存機能が壊れていない

---

## 6. 危険領域チェック

以下に影響する場合は特に慎重に確認する。

* [ ] cards
* [ ] offers
* [ ] trades
* [ ] shipments
* [ ] owner_user_id
* [ ] Trust
* [ ] RLS

該当する場合：

* [ ] ロジックを完全に理解した
* [ ] 意図通りの変更である

---

## 7. コード品質チェック

* [ ] 型エラーがない
* [ ] lintエラーがない
* [ ] 不要なconsole.logがない
* [ ] importが整理されている
* [ ] 可読性が保たれている

---

## 8. 実機確認チェック

* [ ] 実機で確認した
* [ ] UI崩れがない
* [ ] タップ/操作が正常
* [ ] 想定フローがすべて通る

---

## 9. リスクチェック

* [ ] リスクが明記されている
* [ ] 想定外ケースが考慮されている

---

## 10. NGパターン

以下に該当する場合は承認しない。

* [ ] 影響範囲が未記載
* [ ] Source of Truth未確認
* [ ] DB/RLS影響が曖昧
* [ ] 危険領域の理解不足
* [ ] 実機未確認
* [ ] PRの目的が不明確

---

## 11. 承認条件

すべてのチェックを通過した場合のみマージ可能。

それ以外は必ず修正依頼を出す。

---

## 12. 判断原則

迷った場合は以下を優先する。

1. UI正本
2. DB正本
3. 状態遷移
4. 安全性
5. 一貫性

---

## 13. 成功状態

* レビューの質が人によらない
* 見落としが減る
* AIのPR精度が上がる
* 危険領域の事故が減る
