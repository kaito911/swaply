# IMPLEMENTATION FLOW

## 0. 目的

この文書は、SwaplyにおけるAI実装時の標準思考フローを定義する。

AIはこのフローに従ってのみ実装を行うこと。

---

## 1. 前提ルール

* 推測禁止。不明は「不明」と返す
* コードを正としない
* Source of Truth を必ず参照する
* 影響範囲を必ず明示する
* 部分修正は禁止（全文提示）

---

## 2. Source of Truth

### UI正本

* プロトタイプPDF
* 画面仕様書

### 状態正本

* Supabase DB
* 状態遷移設計

---

## 3. 実装フロー

AIは以下の順番を厳守する。

---

### Step 1. タスク理解

* タスク名
* 目的
* 完了条件

を明確化する。

不明点があれば停止する。

---

### Step 2. 対象ファイル特定

* 対象画面（app配下）
* 対象ロジック（lib配下）

---

### Step 3. 関連ファイル探索

必ず以下を確認する：

* 同一機能の既存画面
* 共通コンポーネント
* 型定義
* Supabase通信部分
* ルーティング

---

### Step 4. Source of Truth確認

* UI → プロトタイプと一致しているか
* 状態 → DB構造と一致しているか

---

### Step 5. 影響範囲分析

必ず以下を埋める：

* UI
* DB
* 状態遷移
* RLS
* ルーティング
* 型定義
* 既存機能

---

### Step 6. 危険領域チェック

以下に該当する場合は慎重に扱う：

* cards
* offers
* trades
* shipments
* owner_user_id
* Trust
* RLS

影響する場合は明示する。

---

### Step 7. 修正方針決定

* なぜその実装にするか
* 他案がないか
* 既存との整合性

---

### Step 8. 実装

* 対象ファイルは全文提示
* 関連ファイルも必要なら全文提示
* import整合性を保つ
* 型崩れを許容しない

---

### Step 9. 検証

以下を実行：

```bash
npm run ci:local
```

必要な場合：

```bash
npm run ci:validate
```

---

### Step 10. PR準備

* PULL_REQUEST_TEMPLATE.md をすべて埋める
* 影響範囲を明確にする
* リスクを明記する

---

### Step 11. 停止条件

以下に該当した場合、即停止：

* 仕様不明
* UIと仕様の不一致
* DB変更が必要
* RLS変更が必要
* 状態遷移変更
* owner_user_id 影響
* trades / offers / shipments / Trust 影響

---

## 4. 禁止事項

* 部分コードのみ提示
* UI改変
* DBを見ずに判断
* 影響範囲未確認
* 推測実装

---

## 5. 成功条件

* AIが自律的に実装できる
* 影響範囲を漏れなく把握できる
* 型崩れが発生しない
* CIが通る
* 人間は最終確認のみ
